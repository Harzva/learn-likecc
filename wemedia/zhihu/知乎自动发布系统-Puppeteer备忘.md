# 知乎自动发布系统（Puppeteer）— 备忘与和「官方 API」的对比

> **说明**：正文来自本目录下历史 **GBK 编码**的 `.txt`（文件名在 UTF-8 环境下会显示乱码），已转存为 UTF-8 的 `知乎自动发布系统-从GBK转存正文.txt` 并对照整理。若你本地有 `publish_article.js` 等脚本，请以实际代码为准；本文**不保证**与知乎当前页面 DOM 仍一致。

---

## 一、背景与需求

作为同时运营多平台的创作者，痛点是**内容分发**：同一篇文章要反复调整格式、传图、填标题。核心诉求包括：一键把 Markdown 发到知乎、可选封面、**不依赖知乎开放 API**、能塞进自动化工作流。

---

## 二、技术方案（原文要点）

- **官方 API**：稳，但个人申请门槛高。  
- **纯 HTTP 模拟**：快，但反爬与 Cookie 失效维护成本高。  
- **浏览器自动化（Puppeteer + Stealth）**：模拟真人，相对耐折腾，但依赖 Chrome、资源占用大。

依赖示例：

```bash
npm install puppeteer-extra puppeteer-extra-plugin-stealth puppeteer-core
```

**Cookie**：从创作中心登录后导出关键字段（如 `z_c0`、`SESSIONID`、`d_c0` 等）存 JSON —— **切勿把 Cookie 提交进公开仓库**。

**Draft.js 坑**：标题/正文用 `evaluate` 改 DOM 往往**不计入字数**，需 `click` 聚焦后 `keyboard.type` 分段输入并加延迟模拟真人。

**发布按钮**：class 可能动态变化，可用文案「发布」或可见按钮兜底；发布后需等待「发布中…」结束与跳转。

使用方式（原文列举）：`node publish_article.js ./article.md`、`--config`、`--title/--content/--cover` 等；可配合 crontab 或定时任务。

**注意**：Cookie 会过期、发布频率需节制、内容需过审；失败时依赖 debug 截图排查。

---

## 三、血泪一句：搭半天，不如等平台像飞书钉钉那样开放 API

自己维护 Puppeteer 链路，本质是**和前端改版、风控、Cookie 生命周期**长期对赌：能跑，但**时间成本**往往高于「写内容本身」。对比 **飞书、钉钉** 等产品：**官方开放 API**、文档、配额、Webhook —— 把「发消息 / 写文档 / 建日程」接进工作流，**集成成本可预期**，也不用在每次 UI 改版后改选择器。

对 **知乎、小红书** 等内容平台：若未来有**稳定、合规的创作侧 API**（或官方支持的集成方式），我会**优先**迁过去；在那之前，浏览器自动化只宜当**个人备忘级方案**，不必当成团队标准基建。

---

## 四、与今晚多平台发文的关系

- **知乎 / 小红书人工发**：把 `articles/` 下成稿 Markdown 复制或分段发，仍是最稳、最省心的路径。  
- **自动化脚本**：适合你已经跑通 Cookie 与选择器、且能接受偶发维护的夜晚；**不如**企业协作类产品的开放 API 省心。
