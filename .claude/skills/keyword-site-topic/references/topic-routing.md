# Topic Routing

Use this file when a keyword could fit several areas in this repo.

## Best-fit map

### `site/md/topic-agent-hot.md`

Use for recent Agent ecosystem discussion, product launches, field reports, orchestration patterns, MCP, memory mechanisms, subagent workflows, vendor comparisons, or high-churn external debate that benefits from a curated summary page.

### `site/md/topic-agent-comparison.md`

Use for side-by-side comparison pages where the central value is a structured contrast:

- tool A vs tool B
- product shell vs product shell
- memory system vs memory system
- marketplace vs marketplace

### `site/md/topic-agent.md`

Use when the keyword is broad enough to deserve a stable hub or a new subtopic entry under the Agent umbrella, not just a transient hot take.

### `site/md/topic-rag-hot.md`

Use for RAG news, blog posts, system tricks, benchmark arguments, evaluation debates, or retrieval engineering digests.

### `site/md/topic-rag-papers.md`

Use for paper-focused reading notes or a paper cluster with a research angle.

### `site/md/topic-llm.md`

Use for model internals, architecture taxonomies, inference efficiency, MoE, long-context mechanisms, reasoning cost tradeoffs, and similar model-centric material.

### `site/md/topic-toolchain.md`

Use when the strongest value is workflow practice:

- editor to agent migration
- pricing or channel experience
- usage transparency
- day-to-day tool operating patterns

If the draft becomes highly personal or experiential, consider one of the existing column pages instead of a pure topic page.

### `site/md/topic-websites.md`

Use for evergreen external sites, repos, wikis, personal homepages, or durable reading/bookmark pages rather than fresh news.

### `site/md/topic-personal-knowledge.md`

Use for knowledge accumulation, wiki mindset, long-term memory, personal archive, and "knowledge as system" framing.

### `site/md/topic-sourcemap.md` and `site/md/topic-paoding-jieniu.md`

Use for source-map incidents, codebase reconstruction, product teardown, repo anatomy, or layered architectural interpretation tied to a concrete codebase.

## Fast routing heuristic

Ask:

1. Is this primarily about a fast-moving external discussion?
2. Is this primarily about a stable knowledge area?
3. Is the value in comparison, teardown, or curation?
4. Is the right outcome a hotspot digest, a longform interpretation, or a durable bookmark page?

Use the smallest stable destination that keeps the page easy to find later.

## Image guidance

Prefer one strong visual over many weak ones.

Good visual types for this repo:

- Mermaid flow for process, pipeline, or architecture
- comparison matrix for products or methods
- timeline for releases, incidents, or ecosystem shifts
- screenshot prompt when the UI itself is evidence
- small concept diagram when the article introduces a new mental model
