# AutoFigure：把方法段落变成「能改」的论文配图（附体验站与论文）

> 适合：要写论文方法图、综述示意图、技术博客架构图，又不想在 PPT 里手搓一整天的读者。  
> 体验入口：[https://autofigure.org/zh](https://autofigure.org/zh)  
> 论文（ICLR 2026）：[AutoFigure-Edit: Generating Editable Scientific Illustration](https://arxiv.org/abs/2603.06674)（arXiv:2603.06674）

---

## 一句话说明它在干什么

**AutoFigure** 把「方法描述」这类长文本，尽量自动变成**出版向的科研插图**，而且输出是**结构化 SVG**：文字、箭头、色块、图标往往是**独立对象**，可以在浏览器里继续改，而不是一张改不了细节的位图。

官网中文页写得很直白：结合大模型生成、**SAM3 分割**与**矢量化**，并带可选的**参考图风格迁移**，目标方法论图、系统框图、流程图等都能尝试，统一走「可编辑矢量」这条路线。

---

## 为什么值得单独拎出来说

做技术内容的人（论文、课程、手册）常卡在两件事：

1. **表达**：读者能不能一眼看懂「模块怎么连、数据怎么流」。  
2. **迭代**：审稿人一句「把 Stage 2 画清楚」，你是重画整张 PNG，还是只改两个框？

AutoFigure 的卖点正好压在第二点上：**原生 SVG、组件级可编辑**。这和「生成一张好看的 JPG」不是同一条赛道——后者适合海报感配图，前者更适合**还要改十遍**的学术与工程文档。

官网还强调 **FigureBench** 等评测与长上下文科学理解，方向是「自动化 + 仍能手改」，而不是一次性出图就结束。

## 流水线在干什么（五阶段，便于你对照论文）

根据 [autofigure.org/zh](https://autofigure.org/zh) 公开介绍，整体可以粗分为五步（与论文中的系统设计一致，细节以论文为准）：

| 阶段 | 在做什么 |
|------|-----------|
| 1. 栅格生成 | 视觉语言模型读入方法文本（及可选参考图），先出一张构图草稿（位图），把文字里的结构「翻成」画面布局。 |
| 2. SAM3 分割 | 用 Segment Anything 3 一类能力，把图标、文字区、连线、形状等**拆开**，得到可复用的区域与元数据。 |
| 3. SVG 模板化 | 再经多模态模型生成**对齐分割结果的 SVG 模板**（占位符风格），并结合去背景等步骤整理透明素材。 |
| 4. 最终组装 | 坐标对齐后，用真实图标替换占位符，得到可编辑的 `final.svg` 一类成品。 |
| 5. 迭代优化 | 可选的路径优化、笔画与布局微调；也可在站内 **svg-edit** 画布上继续拖改。 |

如果你熟悉「先粗后细」的制图流程，会把这看成：**生成负责 0→1，分割与矢量化负责 1→可维护资产**。

## 和 DALL·E 类「文生图」差在哪

官网 FAQ 里也提到与通用图像生成器的区别，核心可以记三条：

- **输出形态**：优先 **SVG 矢量**，而不是单张栅格大图。  
- **编辑成本**：改标签、改箭头、改配色，应尽量**局部完成**。  
- **任务假设**：面向**科学插图惯例**（方法框图、pipeline、encoder-decoder 示意等），而不是泛娱乐配图。

若你只为公众号封面要一张氛围图，用通用文生图可能更快；若要**进论文、进幻灯片母版、进可版本管理的源文件**，AutoFigure 这类路线更对口。

## 可以怎么用（结合官网）

1. 打开 [https://autofigure.org/zh](https://autofigure.org/zh)，找到页面上的生成器区域。  
2. **粘贴方法文本**（页面提示：简洁、结构化通常更清晰）。  
3. **可选**：上传参考图做风格迁移（注意站点关于积分、大小的说明）。  
4. 生成后在预览中检查，再进 SVG 编辑做微调。

具体额度、是否开源、支持哪些模型供应商等，以官网 FAQ 与 [论文页面](https://arxiv.org/abs/2603.06674) 的说明为准；论文摘要中亦提到代码与演示链接（见 arXiv 条目中的项目信息）。

## 和本仓库（learn-likecc）的关系说明

本仓库维护的是 **Claude Code 学习向** 的站点与材料（源码导读、课程页、流程图等）。我们在网页里也会用 **Mermaid** 等工具把「Agent 循环、工具调用」画清楚——本质和 AutoFigure 解决的问题同源：**把复杂系统可视化，并尽量可维护**。

不同之处在于：

- **Mermaid / 手写 SVG**：强在工程文档、课程站点的**轻量、可版本控制**。  
- **AutoFigure**：强在**从长科学文本自动起稿**，并落到**可编辑出版向插图**，更贴近论文与正式幻灯片工作流。

二者可以并存：课程站用 Mermaid 讲清楚概念；写论文时再用 AutoFigure 把方法段落收成一版可投的图。
