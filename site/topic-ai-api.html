子专题 模型 API / 聚合

模型 API 不是一类东西
云厂商、聚合层、兼容层要分开看

`七牛云`、`硅基流动`、`OhMyGPT`、`OpenRouter` 看起来都像“一个 Base URL + 一把 Key”，但角色其实很不一样。
这一页的任务，就是先把它们从“都能调模型”里拆开。

Frame

先别比价格，先比“你买的到底是什么”

云厂商模型平台

更像基础设施层，往往还会把存储、媒体、工作流、企业接入一起打包。七牛云在这类里更典型。

一站式 MaaS 聚合

重点是模型丰富度、接入顺滑度和开发体验。硅基流动更像这种“多模型云服务平台”。

OpenAI 兼容网关

强调用一个兼容层去替换上游，代码改动小，路由和计费统一。OhMyGPT 很典型。

跨提供商路由层

重点不只是“能调”，而是 provider 选择、模型目录、请求路由和中立抽象。OpenRouter 是最典型代表。

Four

四家放在一起，该怎么读

01 · 七牛云

七牛云

更接近“云侧 AI 推理平台 + 多媒体能力 + 企业接入”的组合。它的开发者文档已经把 AI Coding 工具的 OpenAI Compatible 配置写得很细，并给出了统一 Base URL https://api.qnaigc.com/v1 的接入方式。

  • 适合本来就有云资源、媒体、存储场景的人
  • 也适合把模型接入和业务底座一起看

AI Coding 配置文档 ↗

02 · SiliconFlow

硅基流动

官方文档把自己定义成“集合顶尖大模型的一站式云服务平台”。这句话很关键，它不是单模型代理，而是强调模型覆盖、按量使用和开发体验。更像开发者友好的 MaaS 平台。

  • 适合想用多种模型，但又不想自己管很多上游账号的人
  • 更像“平台型入口”，而不是纯兼容壳

官方文档 ↗

03 · OhMyGPT

OhMyGPT

它最明确的定位是 OpenAI-compatible API gateway。官方文档直接写明 Base URL、兼容接口，以及“单 API 访问多家模型”的设计。它特别适合老代码平移和各种 CLI / SDK 的低摩擦改造。

  • 适合想尽量少改代码的人
  • 也适合接各种 OpenAI 兼容客户端

官方总览 ↗

04 · OpenRouter

OpenRouter

OpenRouter 最值得看的不是“有多少模型”，而是它把 跨 provider 路由 做成了一等公民。它更适合被理解成“模型交换层 / 路由层”，而不是单纯中转站。

  • 适合经常切模型、切 provider、做 fallback 的团队
  • 也适合拿来做模型实验和路由策略

官方 Quickstart ↗

一句话选型

你要云资源和业务底座一起看：优先看七牛云。

你要一个模型很多、接起来顺的 MaaS 平台：优先看硅基流动。

你要把现有 OpenAI 兼容代码低摩擦迁过去：优先看 OhMyGPT。

你要把多家 provider 当作同一个实验与路由平面：优先看 OpenRouter。
