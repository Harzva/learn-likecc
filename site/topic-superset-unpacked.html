01 · 外层产品壳

Superset 的第一眼就已经说明问题了: 它不是一个 CLI，而是一个完整工作台。顶层同时有 apps/desktop、apps/web、apps/api，旁边还挂着 marketing、docs、streams、mobile 等配套应用。

这意味着它的目标从一开始就不是“把某个 agent 做强”，而是“把很多 CLI agent 收进同一个控制面里”。所以我们读它时，第一层心智模型应该是 工作台壳，不是聊天框。

层关键目录作用
桌面壳apps/desktop把 workspace、pane、terminal、diff/review 装进一个桌面容器里
浏览器 / 服务apps/web、apps/api把同一套工作流延伸到 Web 与服务接口
配套产品面apps/docs、apps/marketing、apps/streams说明这是完整产品体系，不是单个 demo

02 · Pane / Workspace 引擎

packages/panes 是 Superset 很关键的一层。它把布局能力单独抽成一个无头引擎，核心概念直接就是 Workspace、Tab、Pane 与 split layout tree。

这层的价值不是“界面好看”，而是先把 工作位 抽象清楚: 任务在什么 tab，运行在哪个 pane，布局怎么切分，数据和布局如何分离。没有这层，后面的多 agent 调度很容易退化成一堆终端窗口。

概念在 Superset 中的含义对 meta-agent 的价值
Workspace最上层容器，管理多个 tab给“外层 agent 正在管理什么”一个总视图
Tab命名上下文，内部再分 pane把任务切换从“换 app”变成“换工位”
Pane真正承载 terminal/chat/browser 等叶子单元让每个 worker 有稳定可见的落点
Layout tree纯结构树，只描述 split 与权重把布局问题和业务状态拆开，方便扩展

03 · Host-Service 调度中枢

真正像“外层大脑”的，不是桌面 UI，而是 packages/host-service。从 src/app.ts 看，createApp() 一次性接上了数据库、git、workspace filesystem、chat runtime、pull request runtime、event bus、terminal websocket route 和 tRPC router。

也就是说，Superset 真正的 orchestration kernel 是这层服务，而不是 React 组件。前端看到的 pane、terminal、changes、chat，本质上都只是这层 runtime 的可视化窗口。

子系统关键文件职责
App Wiringpackages/host-service/src/app.ts把 db、git、filesystem、chat、terminal、router 统一组装起来
Event Buspackages/host-service/src/events/event-bus.ts把文件系统 / 工作区变化统一广播给前端
Terminal Routepackages/host-service/src/terminal/terminal.ts通过 PTY + WebSocket 管理终端会话的附着与重连
Workspace FSpackages/host-service/src/runtime/filesystem统一工作区文件与路径相关运行时
Control APIpackages/host-service/src/trpc/router把控制面能力暴露给桌面 / Web 前端

04 · Chat Runtime 不是普通聊天框

在 Superset 里，chat 不是一个“发消息的 UI 小组件”，而是一条独立 runtime stack。ChatRuntimeManager 直接围绕 createMastraCode() 的 harness 工作，负责 session 建立、sendMessage、审批、问题响应、plan approval 等控制动作。

这点很关键，因为它说明 Superset 的 chat 实际上承担了 控制协议入口 的角色: terminal 负责执行，chat 负责下达、继续、批准与反馈，host-service 负责把这些状态和动作串成一个系统。

能力体现位置说明
Session RuntimeChatRuntimeManager维护一个 workspace/session 对应的 agent 运行时
控制动作sendMessage、approval、question、plan不只是聊天，而是继续控制内层执行体
错误与显示状态ChatDisplayState把 pending question、error、display state 暴露给前端
与工作区绑定workspaceId、cwdchat 从一开始就不是悬空的，而是绑定执行环境

05 · Superset 如何拼出 Meta-Agent

如果把“meta-agent”理解成 外层 agent 去调用内层 agent 的能力，那 Superset 并不是靠某个神奇模型实现的，而是靠下面这五步结构叠出来的。

步骤工程落点结果
把 inner agent 变成可运行对象terminal / PTY runtime只要能在终端里跑，就能被系统纳入
给每个 agent 一个独立工位workspace + worktree + pane任务隔离，不互相污染
让外层统一看见状态event bus + display state + filesystem/git runtime知道谁在跑、谁卡住、谁已完成
让外层统一控制chat runtime + tRPC可以发送、批准、中断、继续 plan
把结果收回工作台diff/review/editor handoff结果不散落在黑框里，而是回到统一产品流里

所以它的本质不是“多开几个终端”，而是“系统替你管理很多 inner agents”。这就是它的 meta-agent 味道。

06 · 对 Like Code 的启发

我们最该学的，不是照着再做一个 Superset 外壳，而是把其中最值钱的结构拿出来，内生到 Claude Code / Like Code 自己体内。

第一，不要把“终端”当成智能体本体。终端只是观察窗，真正该抽象的是 worker runtime。

第二，Claude Code 要继续扮演主脑。外部 CLI agent 更像 worker，由主线程负责规划、分发、回收与汇总。

第三，产品基线不是口号，而是四个动作: 可拉起、可看见、可控制、可回灌。做到这四件事，才算真正把“调度 agent”做成产品。

一句话收尾: Superset 教我们“外层工作台如何调度 agent”；Like Code 要做的是“让 Claude Code 自己学会调度 agent”。
